# Plan: Fix Multiple WebSocket Connections and Duplicate Queue Joins

## Overview
The root cause of duplicate queue entries is multiple WebSocket connections per user. Currently, the app creates:
- One socket via `socketManager.ts` for game/matchmaking
- Another socket in `useNotifications.ts` for real-time notifications

Each connection has its own `socket.id`, allowing independent `join-three-hands-queue` emissions, resulting in the same user joining multiple times and games with only two distinct players.

## Problem Analysis
- **Root Cause**: Multiple socket connections per user (game + notifications)
- **Impact**: Duplicate queue entries, incomplete games
- **Secondary Issues**: No server-side user-level queue deduplication, potential auth race conditions

## Solution Approach
Fix the root cause first (unify sockets), then add defensive guards.

### Phase 1: Unify Socket Connections
**Goal**: Ensure only one WebSocket connection per user for all features.

1. **Refactor useNotifications to use shared socket**
   **File**: `hooks/useNotifications.ts`
   - Remove standalone `io()` socket creation
   - Import and use `getSocket()` from `socketManager.ts`
   - Listen for notification events on the shared socket
   - Maintain notification-specific connection state tracking

2. **Verify single socket usage**
   - Search codebase for any remaining standalone `io()` calls
   - Ensure all features use the singleton socket
   - Test that only one WebSocket appears in browser network tab

### Phase 2: Add Server-Side Guards
**Goal**: Prevent duplicate joins even if client bugs occur.

1. **Implement user-level queue deduplication**
   **File**: `multiplayer/server/services/UnifiedMatchmakingService.js`
   - Add `isUserInQueue(userId)` method to check existing queues
   - Modify `addToQueue` to reject if `userId` already queued
   - Return clear error message for duplicate join attempts

2. **Add authentication confirmation**
   **File**: `multiplayer/server/socket/handlers/index.js`
   - Modify 'authenticate' handler to emit 'authenticated' event with userId
   
   **File**: `hooks/multiplayer/useSocketConnection.ts`
   - Wait for 'authenticated' event before joining queues
   - Prevent race condition between auth and queue joins

### Phase 3: Testing and Verification
1. **Unit tests**: Verify queue deduplication logic
2. **Integration tests**: Confirm single socket across features
3. **Manual testing**: 
   - Check browser network tab shows only 1 WebSocket
   - Attempt multiple queue joins - should be rejected
   - Verify notifications work on shared socket
   - Test auth confirmation prevents race conditions

## Risk Assessment
- **Low Risk**: Changes are additive and backward-compatible
- **Socket Unification**: Requires careful event listener management
- **Testing**: Critical to verify no broken functionality

## Success Criteria
- Browser shows exactly 1 WebSocket connection per user
- Server rejects duplicate queue joins with user-level check
- Notifications continue working on shared socket
- Authentication confirmation eliminates race conditions
- Games start with correct number of distinct players</content>
