# Socket Architecture Refinement Plan

## Overview
Refactor the server-side UnifiedMatchmakingService into modular services for better maintainability, then implement socket refinements including authentication confirmation, consistent handshakes, and guard rails.

## Current Issues
- Monolithic UnifiedMatchmakingService mixes concerns
- Race condition between authentication and queue joining
- Two-hands uses old broadcastGameStart (no handshake)
- No user-level duplicate queue prevention
- Automatic leave-queue on navigation (undesirable UX)

## Phases

### Phase 1: Server Modularization – Extract QueueManager
**From UnifiedMatchmakingService.js:**
- waitingQueues property
- addToQueue(socket, gameType, userId)
- removeFromQueue(socketId)
- getQueueStatus(gameType)
- cleanupStaleQueues()
- broadcastWaitingUpdate(gameType)

**New file: multiplayer/server/services/QueueManager.js**

### Phase 2: Server Modularization – Extract SocketRegistry
**From UnifiedMatchmakingService.js:**
- socketGameMap property
- gameSocketsMap property
- set(socketId, gameId, gameType, userId)
- get(socketId)
- delete(socketId)
- getGameSockets(gameId)
- cleanupGame(gameId)
- handleDisconnection(socket disconnection logic)
- isUserInQueue(userId) – new method for guard rails

**New file: multiplayer/server/services/SocketRegistry.js**

### Phase 3: Server Modularization – Extract CleanupScheduler
**From UnifiedMatchmakingService.js:**
- startCleanupInterval()
- stopCleanupInterval()
- runCleanup() (calls queueManager.cleanup and socketRegistry.cleanupEmptyGames)

**New file: multiplayer/server/services/CleanupScheduler.js**

### Phase 4: Server Modularization – Extract GameFactory
**From UnifiedMatchmakingService.js:**
- Game type configuration (minPlayers, handSize, etc.)
- createAndRegister(gameType, playerEntries, gameManager, socketRegistry)

**New file: multiplayer/server/services/GameFactory.js**

### Phase 5: Refactor UnifiedMatchmakingService – Pure Orchestrator
**UnifiedMatchmakingService.js becomes:**
- Holds references to QueueManager, SocketRegistry, CleanupScheduler, GameFactory
- Provides same public API (addToQueue, leaveQueue, handleDisconnection, etc.)
- Coordinates calls between modules, no direct manipulation

### Phase 6: Client – Authentication Confirmation
**Problem:** Queue join happens before authentication completes.

**Solution:** Wait for 'authenticated' event confirmation.

**Changes in useSocketConnection.ts:**
- Add authentication state refs and promise
- Emit authenticate, await confirmation
- Only join queues after authentication confirmed

### Phase 7: Client – Remove Automatic Leave-Queue
**Problem:** Automatic leave-queue on navigation forces manual re-queue.

**Solution:** Remove auto leave-queue, add explicit Leave Queue button.

**Changes in app/online-play.tsx:**
- Remove sendLeaveQueueEvent and auto leave-queue useEffects
- Add explicit Leave Queue button in lobby UI

### Phase 8: Server – Unify Two-Hands Game Start
**Problem:** Two-hands uses old broadcastGameStart (no handshake).

**Solution:** Create broadcastTwoHandsGameStart with full handshake.

**Changes:**
- Add broadcastTwoHandsGameStart to BroadcasterService
- Update join-two-hands-queue handler
- Remove old broadcastGameStart method

### Phase 9: Server – Add Guard Rails
**Problem:** Same user can join multiple queues or join while in game.

**Solution:** User-level deduplication checks.

**Changes:**
- Update addToQueue with user checks using socketRegistry.isUserInQueue
- Check if socket already in active game

### Phase 10: Testing
**Test Cases:**
- All game modes create and start correctly
- Disconnections and reconnections work
- No memory leaks from stale data
- Two-hands uses handshake, starts automatically
- No duplicate joins, explicit queue management
- Only one WebSocket connection
