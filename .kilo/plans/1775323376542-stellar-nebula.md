# Plan: Fix Duplicate Queue Joins and Authentication Race Condition

## Overview
The client is sending multiple `join-three-hands-queue` events for the same user, causing duplicate queue entries. This leads to games starting with the same user occupying multiple slots, resulting in incomplete player counts. Additionally, there's a potential race condition where queue joins occur before authentication completes.

## Problem
- **Primary Issue**: Same `userId` can join queues multiple times, bypassing socket-level checks
- **Secondary Issue**: Race condition between authentication emission and queue joining
- **Root Causes**:
  - No server-side user-level queue deduplication
  - Client may emit join events multiple times (re-renders, reconnections, etc.)
  - No authentication confirmation mechanism

## Solution
Implement comprehensive fixes for both issues:

1. **Server-side user-level queue guard**: Prevent same user from joining multiple queues
2. **Client-side deduplication**: Ensure join events aren't sent multiple times
3. **Authentication confirmation**: Wait for auth confirmation before joining queues

## Implementation Steps

### Step 1: Server-side User-level Queue Guard
**File**: `multiplayer/server/services/UnifiedMatchmakingService.js`
- Add `isUserInQueue(userId)` method to check if user is already in any queue
- Modify `addToQueue` to reject if `userId` is already queued
- Return appropriate error response to client

### Step 2: Client-side Join Deduplication
**File**: `hooks/multiplayer/useSocketConnection.ts`
- Strengthen `hasSetupRef` logic to prevent multiple join emissions
- Review `useEffect` dependencies to prevent unnecessary re-runs
- Add additional safeguards against duplicate joins

### Step 3: Authentication Confirmation Flow
**File**: `multiplayer/server/socket/handlers/index.js`
- Modify 'authenticate' handler to emit 'authenticated' event with userId

**File**: `hooks/multiplayer/useSocketConnection.ts`
- Add `isAuthenticatedRef` to track authentication state
- Wait for 'authenticated' event before joining queues
- Handle already-authenticated scenarios on reconnects

## Testing
- Verify no duplicate user entries in queues
- Confirm authentication confirmation prevents race conditions
- Test client-side deduplication across re-renders and reconnections
- Validate error handling for duplicate join attempts

- Low risk: Guards prevent invalid states without breaking valid flows
- Backward compatible: Existing single-join scenarios continue to work
